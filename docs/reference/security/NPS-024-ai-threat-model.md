# NPS-024 — AI Threat Model

## 1. Status of This Document

This document is **normative**. RFC 2119 terms apply as in NPC-001. It is
**Phase 6** of the threat model, covering `TB-AI`. Like Phase 5, this is a
first full pass, not a deepening — Phase 2 (`NPS-020` §2) deferred `TB-AI`
without analyzing it. Unlike Phases 4–5, no AI assistant implementation
exists yet to ground this in real code (checked: no matches for
`assistant`/`CAP-AI` anywhere under `source/`), so this analysis works
directly from `NPS-015`'s specified guarantees, the same method Phase 3
used before any implementation existed.

## 2. Scope

This document covers `SURFACE-AI-0001` (voice control input) and
`SURFACE-AI-0002` (suggestion presentation and confirmation) from NPS-019
§3, examined against `NPS-015`'s suggest-vs-act boundary (§5) — the
mechanism NPC-001 §11.1 and NTM-000 §9 depend on to keep the assistant
from being "the operating system" rather than "a tool."

## 3. Findings

### `FIND-AI-001` — The Suggest-vs-Act Boundary's Security Value Depends on an Unspoofable Confirmation UI, Which Isn't Required to Exist

**What was checked:** whether `NPS-015` §5.2 ("a suggestion MUST be
specific and reviewable... surfaced as an actual toggle the user
activates") or `NPS-009` (Adaptive UI Shell) require that toggle/prompt be
rendered through a UI surface other application containers cannot overlay
or visually mimic.

**What was found:** neither does. `NPS-015` §5.2 requires the suggestion
be *reviewable*, but says nothing about the review surface being *trusted*
— i.e., guaranteed to actually be what it claims to be, rather than a
convincing fake drawn by a different, malicious container.

**Why this matters:** the entire point of §5.1's suggest-vs-act split is
that "the user asked the assistant to do X" and "the user explicitly
authorized this specific action" are treated as different things. That
distinction only has security value if the confirmation the user sees is
actually generated by the assistant and actually reflects the action that
will really execute. A malicious application capable of drawing UI (an
ordinary, common capability — `CAP-DISPLAY` is default-grant per NPS-011
§3) could in principle render a convincing fake "AI suggests: [innocuous
action]" toggle that, when tapped, executes something the fake dialog
never disclosed — a UI-redress/overlay attack, a well-documented class
against exactly this kind of "are you sure" confirmation pattern on other
platforms.

**Severity: High / Medium.** This isn't specific to any implementation
bug — it's a gap in what the specification requires, meaning even a
perfectly-coded assistant following `NPS-015` to the letter today
wouldn't close it.

**Resolution:** `NPS-015` §5.2 amended (§4 below) to require the
confirmation surface be rendered through a protected system-UI path that
ordinary application containers cannot draw over or replicate the
identity of — the same category of guarantee mobile platforms provide for
permission and payment dialogs. New `REQ-AI-0003`.

### `FIND-AI-002` — No Distinct, Auditable Record of What the Assistant Suggested

**What was checked:** whether an AI suggestion and the user's response to
it (approved, declined, ignored) are captured anywhere, distinct from the
capability grant/revocation audit log `NPS-010` §8 and `ADR-0018` already
require.

**What was found:** they aren't. `NPS-010` §8 covers capability events
specifically — a user toggling Bluetooth off through an ordinary settings
action, even one the assistant suggested, isn't a capability grant or
revocation, so it doesn't land in that log at all. There is currently no
answer to "did the assistant suggest this, or did the user decide on
their own?" after the fact.

**Why this matters:** NTM-000 §4 ("Transparency") and the accountability
purpose behind the suggest-vs-act split (§5.1) both lose most of their
value if the record of *what was suggested* doesn't outlive the
conversation it was suggested in.

**Severity: Medium / Medium.**

**Resolution:** `NPS-015` §5 amended (§4 below) to require suggestions and
their outcomes be recorded in the same tamper-evident, hash-chained log
mechanism `ADR-0018` already established for capability events, rather
than inventing a second logging mechanism for a closely related concern.
New `REQ-AI-0004`.

### `FIND-AI-003` — File Search Is Attributed to a Capability That Doesn't Describe File Access

**What was checked:** `NPS-015` §5.3 states file search "MAY be performed
directly by the assistant using `CAP-AI-DIAGNOSTICS-READ`." Cross-checked
against that capability's actual definition in `NPS-011` §3.

**What was found:** `CAP-AI-DIAGNOSTICS-READ` is registered as "read-only
access to system diagnostics and performance counters" — it says nothing
about file content or metadata. File search inherently requires reading
file names, paths, and likely content across the user's storage, which is
a materially broader and more sensitive category of access than
performance counters. `NPS-015` §5.3 is not merely under-specified here;
it cites a capability whose own registry entry doesn't cover what it's
being used to justify.

**Severity: Medium / Medium** (a real spec inconsistency, not yet
exploitable since no implementation exists, but exactly the kind of gap
that becomes a real over-grant the moment one is built against the
current wording).

**Resolution:** `NPS-015` §5.3 amended (§4 below) to require file search
operate under the same ordinary, per-application-scoped filesystem
capability every other application uses (not a special AI-only bypass),
and to stop citing `CAP-AI-DIAGNOSTICS-READ` as covering it.

### `FIND-AI-004` — Code-Assistance File Writes Could Blur Into Persistence Mechanisms

**What was checked:** whether `NPS-015` §5.4's exemption ("code assistance
that produces files... MUST NOT be treated as a system-configuration
change under §5.1 merely because it modifies files") has any boundary
around *where* those files land.

**What was found:** none. As written, the assistant writing to a shell
profile (`.bashrc`), a user-level `systemd` unit, or a cron entry — all
locations an ordinary user-scoped filesystem capability would permit — is
indistinguishable, under §5.4's current wording, from writing an ordinary
document. But those specific locations cause code to run automatically
later, which is functionally a system-behavior change, just one achieved
by "writing a file" instead of a settings toggle — precisely the
distinction §5.1 exists to prevent being routed around.

**Severity: Medium / Low** (requires the assistant to actually write to
one of these specific locations, which nothing currently does, since no
implementation exists — but the gap is in the specification itself).

**Resolution:** `NPS-015` §5.4 amended (§4 below) to exclude
autostart/persistence-mechanism locations from the ordinary-file-write
exemption, routing writes to those specific locations through the same
suggest-and-confirm flow as any other system-configuration change.

### `FIND-AI-005` — Adversarial/Inaudible Voice Commands (Bounded, Not a New Gap)

**What was checked:** whether voice control (`NPS-015` §3, §6.2) has any
exposure to inaudible or adversarially-crafted audio commands — a
documented attack class against voice assistants generally (e.g. commands
encoded outside normal human hearing range).

**What was found:** `NPS-015` doesn't address this directly, but it
doesn't need to: any impactful action a voice command could trigger still
has to cross the suggest-vs-act boundary (§5.1) exactly like a typed one
would. An inaudible command could, at most, cause an unwanted
*suggestion* to appear — which the findings in this document (especially
`FIND-AI-001`, once resolved) already treat as something requiring a
trusted, reviewable confirmation before anything happens.

**Severity: N/A — bounded by controls already required elsewhere, not a
new gap.** Recorded so it isn't re-discovered and mistaken for something
unaddressed later, matching the treatment given to `FIND-BOOT-004`.

## 4. Specification Amendments

**NPS-015** (Local AI Assistant) — four amendments in the same document:
§5.2 requires a protected confirmation UI, closing `FIND-AI-001`; §5
(general) requires suggestion/outcome logging via the existing
`ADR-0018` mechanism, closing `FIND-AI-002`; §5.3 corrected to cite the
right capability model for file search, closing `FIND-AI-003`; §5.4
excludes persistence-mechanism locations from the ordinary-file-write
exemption, closing `FIND-AI-004`.

Applied directly (see its own revision history), per NPS-018 §8.
